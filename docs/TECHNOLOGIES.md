## 1–2. IP, TCP, UDP — фундамент

### IP (Internet Protocol)
- **Что:** доставка пакетов между хостами по `IP`-адресам. Без гарантий —
  пакет может потеряться, продублироваться, прийти не по порядку.
- **Зачем нам:** мы оперируем `SocketAddr` (IP + порт). Адресные типы — в
  `donut-core` (`Address::Ip | Address::Domain`, `Endpoint`).

### TCP (Transmission Control Protocol)
- **Что:** надёжный, упорядоченный байтовый поток поверх ненадёжного IP.
- **Как работает:**
  - **Трёхстороннее рукопожатие**: `SYN → SYN/ACK → ACK` устанавливает
    соединение (и согласует начальные sequence numbers).
  - **Sequence/ACK numbers** нумеруют байты; потерянное переотправляется.
  - **Скользящее окно + congestion control** (cubic/BBR) регулируют темп.
  - **Закрытие**: `FIN/ACK` с каждой стороны (полузакрытие возможно).
- **Важное для прокси:** TCP даёт нам «трубу» байтов — поверх неё работают TLS
  и HTTP/1.1/2. Голый TCP + TLS — это сценарий 1 «Reality» (см. `SCENARIOS.md`).
- **Тюнинг hot-path:** `TCP_NODELAY` (отключить Nagle для低latency),
  `SO_REUSEPORT`, `TCP_FASTOPEN` — через крейт `socket2` (планируется в `donut-io`).
- **Где в коде:** `tokio::net::TcpStream`/`TcpListener`; релей —
  `tokio::io::copy_bidirectional` (см. `donut-server/src/selfsteal.rs`).

### UDP (User Datagram Protocol)
- **Что:** датаграммы без установления соединения, без гарантий и порядка.
  Минимальный заголовок (порт-источник/назначения, длина, контрольная сумма).
- **Зачем нам:** фундамент **QUIC** (а значит HTTP/3 и сценария 3). Также
  `UDP ASSOCIATE` в SOCKS5 для проксирования UDP-приложений (DNS, QUIC игр и т.п.).
- **Почему QUIC на UDP, а не свой протокол на IP:** UDP проходит через NAT и
  файрволы как «обычный» трафик и не требует прав на raw-сокеты.

## 3. Криптопримитивы (слой 4) — кирпичики TLS и REALITY

Эти примитивы используются и внутри TLS, и напрямую в REALITY-метке. Крейты:
`x25519-dalek`, `ring`/`aws-lc-rs`, `aes-gcm`, `hkdf`, `hmac`, `sha2`.

### X25519 / ECDH (обмен ключами)
- **Что:** алгоритм Диффи-Хеллмана на эллиптической кривой Curve25519.
- **Как:** у каждой стороны пара (приватный, публичный). `shared =
  X25519(my_priv, their_pub)` — обе стороны вычисляют **одинаковый** общий
  секрет, не передавая его по сети. Подслушивающий видит только публичные ключи.
- **Зачем нам дважды:**
  1. TLS 1.3 и так делает X25519 в расширении `key_share` для согласования
     ключей сессии.
  2. **REALITY переиспользует тот же эфемерный X25519**, чтобы вывести `AuthKey`
     метки — поэтому на проводе не появляется лишнего ключевого материала.
- **Где:** `donut-veil/src/kx.rs` (`VeilX25519` — кастомная kx-группа поверх
  `x25519-dalek` с **неразрушающим** DH, чтобы один эфемерал дал и TLS-ключи, и
  auth-ключ).

### HKDF (HMAC-based Key Derivation Function, RFC 5869)
- **Что:** из «сырого» секрета (например, ECDH-результат) делает один или
  несколько криптостойких ключей фиксированной длины.
- **Как:** `extract(salt, IKM) → PRK`, затем `expand(PRK, info, L) → ключ`.
  `salt` рандомизирует, `info` привязывает ключ к контексту («доменное
  разделение»).
- **Зачем нам:** `AuthKey = HKDF-SHA256(salt=ClientHello.Random[:20],
  info="REALITY").expand(ECDH)`. TLS 1.3 тоже целиком построен на HKDF
  (key schedule).
- **Где:** `donut-veil/src/auth.rs`.

### AES-GCM (AEAD — Authenticated Encryption with Associated Data, RFC 5116)
- **Что:** шифрование + аутентификация за один проход. Даёт
  конфиденциальность (шифртекст) и целостность (тег): подделка/изменение байтов
  обнаруживается.
- **Ключевые входы:** `key`, `nonce` (число, уникальное на ключ!), `plaintext`,
  `AAD` (данные, которые аутентифицируются, но **не** шифруются).
- **Зачем нам:** REALITY-метка = `AES-256-GCM seal` 16-байтного plaintext, где
  **AAD = весь ClientHello с занулённым SessionID**. Это связывает метку с
  конкретным ClientHello (нельзя переклеить в другой). TLS 1.3 шифрует записи
  тем же классом AEAD.
- **Почему nonce уникален:** повтор nonce на одном ключе ломает GCM
  катастрофически. У нас nonce = `ClientHello.Random[20..32]` (рандом клиента).
- **Где:** `donut-veil/src/auth.rs` (`seal`/`open`).

### HMAC / SHA-2
- **HMAC** — код аутентичности сообщения на основе хеша и ключа: «подписал тот,
  кто знает ключ». REALITY проверяет `HMAC-SHA512(AuthKey, cert.signature)`,
  чтобы клиент убедился, что нет MITM.
- **SHA-256/512** — криптохеши; основа HKDF/HMAC.
- **Где:** `donut-veil/src/auth.rs`, крейты `hmac`, `sha2`.

## 4. TLS 1.3 (слой 5) — приватность поверх TCP (RFC 8446)

### Что и зачем
TLS даёт три вещи: **конфиденциальность** (шифрование), **целостность** (защита
от изменения), **аутентификацию** (сертификат сервера). Для нас TLS — ещё и
**холст для маскировки**: REALITY встраивается прямо в TLS-рукопожатие.

### Структура: Record Layer
Всё в TLS едет «записями» (records):
```
тип(1) | версия(2) | длина(2) | payload(длина)
0x16 = handshake, 0x17 = application_data, 0x14 = change_cipher_spec, 0x15 = alert
```
Первый ClientHello — это record с типом `0x16`. Именно его читает наш
selfsteal-триаж (`donut-server/src/selfsteal.rs`), снимает 5-байтовый заголовок
и отдаёт «тело» (handshake-сообщение) в `donut-veil::server_verdict`.

### Рукопожатие TLS 1.3 (1-RTT)
```
Client                                Server
  ClientHello  ───────────────────────►        (key_share, SNI, ALPN, versions, SessionID)
                                       ◄─────── ServerHello (key_share)
                                       ◄─────── {EncryptedExtensions, Certificate,
                                                  CertificateVerify, Finished}
  {Finished}   ───────────────────────►
  [application_data]  ◄──────────────►  [application_data]
```
- После `ServerHello` всё уже зашифровано (в отличие от TLS 1.2).
- **1-RTT**: данные можно слать после одного round-trip. Есть **0-RTT**
  (ускорение для возобновляемых сессий) — мы его не используем (риски replay).

### Ключевые поля ClientHello (важны для маскировки)
- **`SNI`** (Server Name Indication) — имя хоста **открытым текстом**. DPI читает
  его, чтобы понять, к какому сайту идём. У нас SNI = домен подложки.
- **`ALPN`** — какой протокол поверх TLS (`h2`, `http/1.1`, `h3`). Должен
  совпадать с тем, что реально умеет подложка.
- **`key_share`** — эфемерный публичный X25519 клиента (его и переиспользует REALITY).
- **`legacy_session_id` (SessionID, 32 байта)** — в TLS 1.3 семантически не нужен
  (наследие TLS 1.2). **REALITY прячет в него зашифрованную метку.** Для зонда
  это поле выглядит как случайные 32 байта — норма.
- **Порядок расширений / GREASE** — формируют **fingerprint** ClientHello. У
  каждого TLS-стека он свой; rustls по умолчанию узнаваемо «не браузерный».
  Поэтому нужен **uTLS** — имитация Chrome/Firefox (планируется через форк, M7).

### Почему форк rustls (`crates/donut-tls`)
Mainline rustls не даёт хуков, нужных REALITY: (1) мутировать SessionID в
ClientHello на клиенте после сериализации, (2) на сервере увидеть **сырые**
байты ClientHello до криптообработки и решить tunnel/forward. Мы форкнули rustls
(пакет назван `rustls`, подключён через `[patch.crates-io]`, физически в
`crates/donut-tls`) и добавили:
- `ClientConfig::client_hello_mutator` — клиентский хук;
- `ServerConfig::raw_client_hello_hook` + `VeilDecision{Tunnel,Forward}`;
- `ServerConnection::take_forwarded()` — забрать сырой ClientHello для релея.
Подробности патч-точек — см. `crates/donut-tls` и `donut-rustls` (тонкая обёртка-фасад).

## 5. QUIC (слой 6) — надёжный транспорт поверх UDP (RFC 9000)

### Что
QUIC — это «TCP + TLS 1.3, переосмысленные», но поверх **UDP** и в user-space.
Основа HTTP/3 и нашего сценария 3.

### Как работает (ключевые идеи)
- **Встроенный TLS 1.3:** рукопожатие QUIC = рукопожатие TLS 1.3, перенесённое в
  QUIC-фреймы (`CRYPTO`). Отдельного «TLS поверх» нет — он внутри.
- **Мультиплекс потоков без head-of-line blocking:** много независимых
  `stream`-ов в одном соединении; потеря в одном не тормозит другие (в отличие от
  HTTP/2-над-TCP, где потеря TCP-сегмента стопорит все потоки).
- **Connection ID:** соединение живёт не по `(IP,порт)`, а по ID — переживает
  смену сети (роуминг Wi-Fi↔LTE).
- **0-RTT / 1-RTT** установление; быстрее TCP+TLS.
- **Datagram-расширение** (RFC 9221) — ненадёжные датаграммы поверх QUIC (удобно
  для проксирования UDP).
- **Congestion control** в user-space (cubic/BBR/кастом — см. §10).
- **Шифрование почти всего заголовка** → DPI видит мало; зато сам факт «это QUIC»
  заметен (UDP/443, форма Initial-пакета). Поэтому маскировка — «выглядеть как
  обычный HTTP/3-сайт» (как у Hysteria).
- **REALITY в QUIC:** так как рукопожатие — это TLS 1.3, тот же `rustls`-форк
  (через `quinn`) даёт REALITY-хуки и на QUIC.
- **Где:** `donut-quic` (`quinn` 0.11). Сейчас: H3 stream-one и raw-bidi.

## 6. HTTP/1.1, HTTP/2, HTTP/3 (слой 7)

XHTTP (наш транспорт) умеет ехать поверх всех трёх. Почему важно понимать
различия — потому что от версии зависит, как выглядит маскировка.

### HTTP/1.1
- Текстовый, по одному запросу-ответу на TCP-соединение (keep-alive переиспользует).
- Простой, проходит везде; легко имитировать «обычный POST/GET».
- Крейт: `hyper` (h1).

### HTTP/2
- Бинарный, **мультиплекс** многих запросов в одном TCP-соединении (фреймы
  `HEADERS`/`DATA`, stream id), сжатие заголовков (HPACK), server push (почти не
  используется).
- Минус: **head-of-line blocking на уровне TCP** (потеря сегмента стопорит все
  стримы). Лечит это HTTP/3.
- Крейт: `hyper` (h2).

### HTTP/3
- HTTP поверх **QUIC** (см. §5). Мультиплекс без HoL-blocking, сжатие QPACK,
  ALPN = `h3`.
- Для нас — транспорт сценария 3 (низкая латентность, lossy-сети).
- Крейты: `h3`, `h3-quinn`.

### Зачем XHTTP «притворяется HTTP»
Прокси-трафик упакован в обычные с виду `POST`/`GET` запросы (часто с
`Content-Type: application/grpc` и padding через `Referer`) — для DPI это похоже
на нормальное веб-приложение, особенно за CDN. Байт-детали — `PROTOCOLS.md §3`.

## 7. Прокси-обвязка (слой 8)

### SOCKS5 (RFC 1928) — клиентский inbound
- **Что:** стандартный протокол «прокси-листенер»: приложение говорит SOCKS5-прокси
  «соедини меня с `host:port`», прокси делает это от его имени.
- **Поток:** greeting (методы аутентификации) → выбор метода → `CONNECT`/`UDP
  ASSOCIATE` запрос с адресом → ответ → дальше прозрачный байтовый поток.
- **Зачем нам:** `donut-client` поднимает SOCKS5 локально (`127.0.0.1:1080`);
  браузер/система ходит через него, а клиент заворачивает это в туннель к серверу.
- **Где:** `donut-socks` (сейчас NO-AUTH + CONNECT), используется в
  `donut-client/src/local_proxy.rs`.

### HTTP CONNECT inbound
- Альтернативный клиентский listener: метод `CONNECT host:port` HTTP-прокси.
  Планируется в M7.

### DNS / DoH
- **Зачем:** резолвить доменные имена в IP (и желательно приватно, чтобы DPI не
  видел запросы). **DoH** (DNS over HTTPS) прячет DNS в обычный HTTPS.
- **Где:** `donut-dns` (план: system UDP + DoH `https://1.1.1.1/dns-query`),
  крейт `hickory-resolver`. Сейчас — заглушка.

### geoip / geosite
- **Что:** базы соответствий «IP-диапазон → страна» и «домен → категория»
  (формат v2fly `.dat`, protobuf).
- **Зачем:** роутинг/split-tunnel: например, `geoip:cn` и локальные домены — мимо
  туннеля (direct), остальное — в туннель.
- **Где:** `donut-geo` (план: переиспользуем `geosite-rs`). Сейчас — заглушка.

### Routing
- Сопоставление `domain/ip/port/user → outbound tag`. Движок матчинга в
  `donut-routing` (заглушка), потребитель — `donut-server`/`donut-client`.

## 8. Прикладная маскировка (слой 9) — VLESS · REALITY · XHTTP · Vision

Это «верх» стека — то, ради чего всё остальное. Здесь только суть; байт-спеки —
в [`PROTOCOLS.md`](PROTOCOLS.md), REALITY/Self-Steal — в
[`REALITY-SELFSTEAL.md`](REALITY-SELFSTEAL.md).

### VLESS — тонкая прокси-обёртка
- **Что:** минимальный протокол поверх (уже зашифрованного) транспорта: «кто ты
  (UUID), куда соединить (адрес+порт), как (flow)». Без своего шифрования —
  полагается на TLS/REALITY снизу.
- **Почему он, а не VMess:** VLESS без своей крипты → нет timing-утечек VMess,
  меньше кода, отлично сочетается с REALITY+XHTTP.
- **Где:** `donut-wire` (encode/decode заголовка, валидатор flow).

### REALITY — маскировка под чужой/свой реальный сайт
- **Что:** прячет метку «свой» в `SessionID` ClientHello; чужих прозрачно
  проксирует на реальный сайт (`dest`). Защита от активного зондирования.
- **Где:** `donut-veil` (auth + вердикт), `donut-server/src/selfsteal.rs`
  (триаж + relay). Полный разбор — `REALITY-SELFSTEAL.md`.

### XHTTP — транспорт, маскирующийся под обычный HTTP
- **Что:** три режима (`stream-one`/`stream-up`/`packet-up`) поверх H1/H2/H3;
  прокси-данные едут как тело обычных POST/GET. `stream-one` — дефолт под REALITY.
- **Где:** `donut-carrier` (H1/H2), `donut-quic` (H3).

### Vision (`xtls-rprx-vision`) — лечение TLS-in-TLS
- **Что:** убирает сигнатуру «двойного TLS», сплайся внутренние TLS-записи
  напрямую после детекта окончания внутреннего рукопожатия. Только raw-TCP+REALITY.
- **Где:** план — `donut-wire` (padding/`trafficState`), milestone M5.5. Байты —
  `PROTOCOLS.md §4`.

## 9. Congestion control (управление перегрузкой) — §10 для QUIC

- **cubic** — дефолт, loss-based: на потерю пакета снижает окно. Дружелюбен к
  соседям, но проседает на lossy/throttled каналах.
- **BBR** — model-based: оценивает полосу и RTT, не паникует на единичных
  потерях. xray использует BBR для H3 c v26.3.27.
- **Brutal (Hysteria-style)** — задаёт **фиксированную целевую полосу** и шлёт с
  ней, игнорируя loss-backoff. Агрессивно «пробивает» троттлинг/потери ценой
  «недружелюбности». У нас — **опциональная** надстройка для сценария 3
  (см. `PLAN.md`, M5). Не дефолт.

## 10. Как слои собираются в наши сценарии

```
Сценарий 1 «Reality»:   App → VLESS → Vision → TLS1.3(REALITY) → TCP → IP
Сценарий 2 «XHTTP»:     App → VLESS → XHTTP → HTTP/1.1|2 → TLS1.3(REALITY) → TCP → IP
Сценарий 3 «QUIC/H3»:   App → VLESS → XHTTP → HTTP/3 → QUIC(TLS1.3 REALITY) → UDP → IP
```
Что именно видит наблюдатель в каждом случае и почему это незаметно — разобрано
в [`SCENARIOS.md`](SCENARIOS.md).
