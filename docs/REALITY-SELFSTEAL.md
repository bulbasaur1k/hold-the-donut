# REALITY и Self-Steal — как прокси прикидывается обычным сайтом

> Учебно-справочный документ. Цель — объяснить «почему так», а не только
> «как в коде». Ground-truth по байтам — в [`PROTOCOLS.md`](PROTOCOLS.md) §2/§4;
> решения по сценариям и подложке — в [`PLAN.md`](PLAN.md).
> Эталон: xray-core v26.4.15.

## 0. С чего начать: модель угрозы

Противник — государственный файрвол (GFW и аналоги). У него три инструмента:

1. **Пассивный DPI** — смотрит на проходящий трафик и ищет сигнатуры
   (паттерны байтов, размеры записей, тайминги). Дёшево, работает на лету.
2. **Активное зондирование (active probing)** — сам подключается к
   подозрительному `IP:port` и проверяет, «честный» ли это сервис. Если на
   `:443` отвечает что-то, что не похоже на настоящий веб-сайт, — IP в бан.
3. **Статистика/корреляция** — объём, длительность, периодичность соединений.

Вывод, из которого растёт REALITY: **мало зашифровать трафик — сервер обязан
при прямом обращении выглядеть как настоящий, ничем не примечательный сайт.**

## 1. Краткая эволюция маскировки (чтобы понять, зачем REALITY)

| Поколение | Идея | Чем убили |
|---|---|---|
| Domain fronting | SNI=невинный домен CDN, Host=реальный | CDN-провайдеры запретили несовпадение SNI/Host |
| TLS + поддельный SNI (старый Reality-подобный) | притворяемся, что идём на популярный домен | зонд приходит с тем же SNI → видит, что сертификат «не тот» / поведение отличается |
| **TLS-in-TLS туннели** (Trojan, VLESS+TLS) | заворачиваем трафик в честный TLS к своему домену | **двойной TLS** оставляет сигнатуру: вложенные TLS-записи имеют узнаваемый паттерн длин/таймингов (см. Vision, §6) |
| **REALITY** | крадём настоящий TLS-хендшейк реального сайта; для чужих — прозрачно проксируем туда же | актуально; ломается только при ошибках конфигурации (см. §5, §7) |

## 2. REALITY в двух абзацах

REALITY встраивается **в сам TLS 1.3 ClientHello**, не добавляя на провод
ничего, что отличало бы его от обычного браузера. Клиент и сервер заранее
обменялись через конфиг парой ключей X25519 (у сервера — приватный, у клиента —
публичный сервера) и коротким идентификатором `shortId`.

Клиент кладёт в **legacy-поле `SessionID`** ClientHello (32 байта, которые в
TLS 1.3 семантически не используются) зашифрованную «метку свой/чужой». Сервер
читает её ещё **до** криптографии TLS. Сошлось — это наш клиент, поднимаем
туннель. Не сошлось (зонд, браузер, мусор) — **молча и байт-в-байт проксируем
соединение на реальный сайт** (`dest`/`target`). Зонд видит честный сертификат и
настоящий контент того сайта и не может отличить нас от обычного reverse-proxy.

## 3. Механика «метки» (auth) по шагам

Ключи и деривация (байты — в [`PROTOCOLS.md` §2](PROTOCOLS.md)):

```
shared  = X25519(client_eph_priv, server_static_pub)   // тот же ECDH,
          = X25519(server_static_priv, client_eph_pub)  // что и в TLS 1.3 key_share
salt    = ClientHello.Random[0..20]                     // 20 байт
AuthKey = HKDF-SHA256(salt, info="REALITY").expand(shared)   // 32 байта
```

Хитрость: REALITY **переиспользует тот самый эфемерный X25519**, который клиент
и так кладёт в расширение `key_share` для TLS 1.3. Поэтому на проводе не
появляется лишнего ключевого материала — только обычный ClientHello.

Что лежит внутри `SessionID` (32 байта) — это **AES-256-GCM seal** 16-байтного
открытого текста:

```
plaintext (16 B): version(3) | reserved(1=0) | unix_ts(4, BE) | shortId(8)
nonce          : ClientHello.Random[20..32]        (12 байт)
AAD            : весь ClientHello, но слот SessionID занулён
ciphertext+tag : 16 + 16 = 32 байта → кладём в SessionID[0..32]
```

Симметрия AAD (обе стороны зануляют SessionID перед расчётом) — то, что делает
проверку воспроизводимой на сервере. `unix_ts` даёт защиту от грубого replay
(сервер может отбраковать слишком старые метки).

Зачем `shortId`: один сервер обслуживает несколько клиентов/тарифов; `shortId`
позволяет различать их (и при желании отзывать), не раздавая разные ключи.

## 4. Развилка сервера: Tunnel vs Forward

```
        ClientHello прилетел на :443
                  │
       читаем SessionID[0..32]  (ещё до TLS-крипто!)
                  │
        AES-GCM Open(AuthKey, nonce, AAD)
            ┌─────┴─────┐
         успех        провал
            │             │
   shortId в наборе?    Forward → dest        (зонд / браузер / мусор)
       ┌───┴───┐
      да       нет → Forward → dest           (открылось, но клиент чужой)
       │
   TUNNEL: завершаем TLS и проксируем туннель
```

Ключевое свойство **Forward**: он **байт-прозрачен**. Сервер НЕ терминирует TLS
для чужого — он лишь пересылает уже прочитанные байты (включая сам ClientHello) и
дальше тупо копирует поток в обе стороны. Поэтому зонд устанавливает TLS
напрямую с `dest` и видит его настоящий сертификат и контент.

## 5. Две стратегии `dest`: Borrowed и Self-Steal

| | **Borrowed** (одолженный) | **Self-Steal** (свой) |
|---|---|---|
| `dest` | внешний популярный сайт (`www.microsoft.com:443`) | свой веб-сервер на той же машине (`127.0.0.1:8443`) |
| SNI | чужой домен | твой домен |
| Сертификат для зонда | настоящий чужого сайта | настоящий твоего домена (Let's Encrypt) |
| Латентность Forward | до внешнего сайта | localhost |
| Минусы | твой ASN/PTR/гео ≠ чужой домен; чужой сайт обязан держать TLS1.3+X25519+H2 и не быть с тобой в одной CDN; редиректы/гео-различия зонд может заметить | домен и контент целиком на тебе → **нужна достоверная подложка** |

**Self-Steal** — это «украсть у самого себя»: вместо чужого сертификата ты
подставляешь **свой реальный** сайт с валидным сертификатом. Консистентно по
ASN/PTR/домену, без зависимости от чужого аптайма, localhost-латентность. Это наш
рекомендуемый дефолт (см. [`PLAN.md`](PLAN.md) → «Self-Steal и подложка»).

## 6. Подложка (decoy/backdrop): что значит «достоверный сайт»

Пустая дефолтная страница nginx, `444`/обрыв или голый `200 OK` — палево.
Требования:

- реальный контент: несколько страниц, ассеты, favicon, `robots.txt`;
- консистентные заголовки (`Server:`, HSTS, тип контента); распространённый
  TLS-стек;
- **ALPN совпадает** с тем, что подложка реально умеет (h2/http1.1) — forward
  байт-прозрачный, поэтому ALPN определяет именно `dest`;
- **одинаковое поведение для зонда и для браузера** — никаких веток «если не наш
  — отдать другое»;
- правдоподобная причина трафика (страница загрузок/облако/статус — объясняет
  полосу).

Варианты реализации (детали и наши deliverables — в `PLAN.md`):
внешний nginx/caddy рядом (прод-дефолт) · встроенный статик-HTTPS decoy
(feature `decoy`) · генератор `donut-tools gen-decoy` · (не рекомендуем)
reverse-proxy-зеркало.

## 7. Vision и проблема TLS-in-TLS (зачем сценарий «Reality»)

Даже внутри честного TLS-туннеля есть утечка: если по туннелю идёт **вложенный
HTTPS**, наблюдатель видит две вложенные последовательности TLS-записей.
Соотношение длин/таймингов внешних и внутренних записей — устойчивая сигнатура
прокси. `xtls-rprx-vision` это лечит: он осматривает первые пакеты, ловит момент
окончания **внутреннего** TLS-хендшейка и затем перестаёт заворачивать —
**сплайсит** соединение, чтобы внутренние TLS-записи ехали напрямую. Подробная
байт-механика (padding-команды `0x00/0x01/0x02`, `trafficState`, переход в
direct-copy) — в [`PROTOCOLS.md` §4](PROTOCOLS.md). Vision работает только по
raw-TCP + (REALITY|TLS) и несовместим с XHTTP (issue #5576).

## 8. Как это сделано у нас (карта кода)

```
TCP :443 ──► donut-server: selfsteal::triage()
                │  читает первую TLS-запись (ClientHello), не теряя байты
                ▼
        donut-veil::server_verdict(&VeilServerConfig, client_hello)
                │   (та же логика, что и в rustls-хуке, но без TLS-машины)
            ┌───┴────────────────┐
       Verdict::Forward      Verdict::Tunnel
            │                     │
   relay_to_dest():          Triage::Tunnel{client, prefix}
   connect(dest);            → (далее) терминируем TLS и отдаём в carrier
   write_all(prefix);          (M6 step2 tunnel-ветка / M7 step2)
   copy_bidirectional()
   = байт-прозрачный selfsteal
```

- `donut-veil/src/server.rs` — `decide()` (общее ядро), `server_verdict()`
  (транспортный API без `rustls`), `build_raw_client_hello_hook()` (для пути
  через форк rustls). Байт-разбор ClientHello — `parse.rs`, крипто — `auth.rs`.
- `donut-server/src/selfsteal.rs` — `triage()`: читает ClientHello,
  спрашивает вердикт, на `Forward` прозрачно релеит на `dest`.
- Форк rustls (`crates/donut-tls`, package name `rustls`, подключён через
  `[patch.crates-io]`) — хуки `client_hello_mutator` / `raw_client_hello_hook` /
  `VeilDecision` / `ServerConnection::take_forwarded()`.

## 9. Чек-лист устойчивости к активному зондированию

- [ ] `dest` отдаёт настоящий валидный сертификат для SNI-домена;
- [ ] ALPN сервера-подложки реально поддерживается (h2/http1.1);
- [ ] поведение идентично для зонда и для браузера (нет ветвления по «свой/чужой»
      на HTTP-уровне — ветвление только на TLS-уровне и невидимо);
- [ ] нет дефолт-страницы nginx / пустого 200 / обрыва;
- [ ] TLS 1.3 only; TLS 1.2 ClientHello → Forward;
- [ ] метки с устаревшим `unix_ts` отбраковываются (anti-replay);
- [~] фингерпринт ClientHello клиента не выдаёт rustls по JA3: режим
      `randomized` рандомизирует порядок cipher suites/extensions
      (`donut-veil/fingerprint.rs`, см. [FINGERPRINT.md](FINGERPRINT.md)).
      Parrot-пресеты (Chrome/Firefox) и GREASE требуют расширения хука
      форка и отложены (M7/M10).

## 10. Глоссарий

- **Active probing** — активное зондирование: файрвол сам стучится на сервер.
- **SNI** — Server Name Indication, имя хоста в открытом виде в ClientHello.
- **ALPN** — согласование протокола поверх TLS (h2, http/1.1, h3).
- **`dest`/`target`** — куда REALITY проксирует чужих (подложка).
- **selfSteal** — `dest` = свой сайт на той же машине.
- **`shortId`** — короткий идентификатор клиента в метке REALITY.
- **AAD** — Additional Authenticated Data в AEAD (тут — ClientHello с занулённым
  SessionID).
- **TLS-in-TLS** — туннелирование TLS внутри TLS; источник сигнатуры, лечится
  Vision.
