## 0. Что значит «скрываться» — модель наблюдателя

У файрвола два режима (подробно — `REALITY-SELFSTEAL.md §0`):

1. **Пассивный DPI** — смотрит проходящий трафик: SNI, размеры/тайминги записей,
   ALPN, форму пакетов. Цель маскировки — выглядеть как массовый легальный
   трафик (обычный HTTPS-сайт, обычный HTTP/3-сайт).
2. **Активный зонд** — сам подключается к нашему `IP:443` (TCP или UDP) и
   проверяет, «честный» ли сервис. Цель — при прямом обращении отдать настоящий
   сайт (это делает REALITY-Forward / Self-Steal).

**Общий знаменатель всех трёх сценариев:** ядро `VLESS + REALITY`. REALITY
отвечает за оба пункта: метка «свой» в ClientHello (для пассивного DPI ничего не
торчит) и прозрачный Forward чужих на подложку (для активного зонда). Различие
сценариев — в **транспорте поверх REALITY** и в том, на что именно мы похожи.

## Сценарий 1 — «Reality» (VLESS + REALITY + Vision, raw TCP)

```
App → VLESS → Vision(xtls-rprx-vision) → TLS 1.3 (REALITY) → TCP → IP
```

### Как устанавливается
1. Клиент шлёт обычный TLS 1.3 ClientHello на `:443`, но с REALITY-меткой в
   SessionID и (в идеале) uTLS-фингерпринтом браузера. SNI = домен подложки.
2. Сервер читает метку до крипто: свой → tunnel; чужой → Forward на `dest`.
3. В tunnel внутри TLS идёт VLESS-заголовок, затем — данные. **Vision** после
   детекта окончания внутреннего TLS-рукопожатия перестаёт оборачивать и
   сплайсит внутренние TLS-записи напрямую (нет «двойного TLS»).

### Что видит пассивный DPI
- Один TLS 1.3 поток на `:443` к «домену подложки».
- Без Vision был бы виден паттерн **TLS-in-TLS** (вложенные записи) → Vision его
  убирает. С Vision поток выглядит как одно обычное TLS-соединение к сайту.

### Что видит активный зонд
- Постучавшись на `:443`, получает **настоящий** TLS-хендшейк и контент подложки
  (Self-Steal: свой сайт с валидным сертификатом). Неотличимо от reverse-proxy.

### Плюсы / минусы
- ➕ Самый быстрый: нет HTTP-обёртки, нет двойного шифрования (Vision), минимум
  оверхеда. ➕ Минимальная сигнатура.
- ➖ Голый TCP/TLS — не «прячется в CDN»; ➖ Vision требует аккуратной реализации
  (padding/splice). ➖ Не пройдёт там, где режут весь не-CDN TLS на 443.

### Когда применять
Свой VPS с белым IP и доменом, когда нужна максимальная скорость и минимальная
заметность, а блокировки — по DPI/зондированию, не по «всё кроме CDN».

### Статус
Milestone **M5.5** (Vision поднят из M10-опционального в core). Байты — `PROTOCOLS.md §4`.

## Сценарий 2 — «XHTTP» (VLESS + REALITY + XHTTP, H1/H2)

```
App → VLESS → XHTTP(stream-one|stream-up|packet-up) → HTTP/1.1|HTTP/2 → TLS 1.3 (REALITY) → TCP → IP
```

### Как устанавливается
1. Тот же REALITY-ClientHello → tunnel/forward.
2. Внутри TLS — обычные HTTP-запросы: прокси-данные едут как тело `POST`/`GET`
   (часто `Content-Type: application/grpc`, padding через `Referer`). `stream-one`
   несёт оба направления в одном запросе (дефолт под REALITY).

### Что видит пассивный DPI
- Похоже на нормальное HTTPS-веб-приложение: TLS на 443 + внутри валидный HTTP/2.
- **Дружит с CDN:** можно поставить за Cloudflare/любой CDN — трафик сливается с
  морем обычных сайтов за тем же фронтом.

### Что видит активный зонд
- То же, что и в сценарии 1: REALITY-Forward отдаёт реальный сайт подложки.
  (Если за CDN — зонд упирается в CDN, что ещё незаметнее.)

### Плюсы / минусы
- ➕ Максимальная «вписанность» в веб-трафик, CDN-фронтинг, переживает
  reverse-proxy. ➕ Три режима под разные каналы.
- ➖ Оверхед HTTP-обёртки и (в H2) head-of-line blocking на потерях. ➖ Чуть
  медленнее сценария 1.

### Когда применять
Жёсткая цензура, где выживает только «трафик как у больших сайтов»/за CDN; мобильные
сети; когда важнее незаметность, чем последний процент скорости.

### Статус
Milestone **M4 ✅** (stream-one/stream-up/packet-up над H1/H2). Байты — `PROTOCOLS.md §3`.

## Сценарий 3 — «QUIC / HTTP-3» (Hysteria-like)

```
App → VLESS → XHTTP → HTTP/3 → QUIC (TLS 1.3 REALITY) → UDP → IP
```

### Как устанавливается
1. QUIC-рукопожатие = TLS 1.3 внутри QUIC-`CRYPTO`-фреймов → тот же REALITY-хук
   (через `quinn` + форк rustls).
2. Поверх — HTTP/3 (ALPN `h3`), внутри — XHTTP, как в сценарии 2, но без
   HoL-blocking и с UDP-латентностью.
3. Опционально — **Brutal congestion control** (фиксированная полоса) для
   lossy/throttled сетей (идея от Hysteria).

### Что видит пассивный DPI
- UDP/443 с формой QUIC Initial → «обычный HTTP/3-сайт». Большая часть заголовка
  QUIC зашифрована, читать почти нечего. Заметен сам факт «это QUIC к 443».

### Что видит активный зонд
- Постучавшись QUIC-ом, неавторизованный получает поведение реального HTTP/3-сайта
  (REALITY-Forward на QUIC-handshake — masquerade, аналог selfsteal). Свой
  собственный H3-auth/reverse-proxy/Salamander мы **не** делаем (interop с xray важнее).

### Чем это «аналог Hysteria», а чем нет
- ➕ Берём: QUIC-транспорт, низкую латентность, идею **Brutal-CC** и masquerade
  неавторизованных.
- ➖ Не берём: это **не** отдельный протокол Hysteria. Остаёмся **xray-совместимым**
  XHTTP-over-H3; не делаем собственный auth, Salamander-обфускацию и H3-reverse-proxy.

### Плюсы / минусы
- ➕ Низкая латентность, устойчивость к потерям/роумингу сети (QUIC connection id),
  обход TCP-троттлинга, опц. Brutal под плохие каналы.
- ➖ Некоторые сети режут/деприоритизируют UDP/443 → тогда фолбэк на сценарий 2.

### Когда применять
Lossy/высоколатентные/троттлящие каналы (мобильный интернет, дальние линки), где
TCP проседает; когда UDP/443 не зарезан.

### Статус
Milestone **M5** (H3 stream-one ✅, raw-bidi ✅; H3-framing на raw-bidi — pending;
Brutal-CC — опц. надстройка). Байты — `PROTOCOLS.md §3` (та же модель XHTTP, транспорт H3).

## Выбор сценария — короткое правило

- Свой VPS, нужна скорость, блок по DPI/зонду → **Reality (1)**.
- Жёсткая цензура / нужен CDN / мобилки → **XHTTP (2)**.
- Плохой/троттлящийся канал, UDP жив → **QUIC/H3 (3)**, при необходимости с Brutal-CC.
- На клиенте разумно держать **фолбэк-цепочку 3 → 2** (если режут UDP).
